# 한국코드페어 심사기준 자가점검

기준: SW공모전 애플리케이션 또는 실생활에 유용한 도구 개발 유형의 공개 심사기준 구조

## 1. 개발 목적 / 20점

상태: 강함

문서 반영 위치:

- 작품요약서: 첫 문단
- 작품설명서: `1. 개발 배경 및 필요성`, `2. 개발 주제와 목적`

강점:

- 반복 오답과 개념 이해 부족을 실제 학습 문제로 정의했다.
- 코드페어 주제인 AI와 데이터 활용을 첫 문단에서 바로 연결했다.
- AI API 실패와 생성 문항 검수 문제를 제약으로 인정했다.

남은 리스크:

- 실제 학생 장기 데이터가 아직 적다. 발표에서는 장기 효과를 증명했다고 말하지 않는다.

## 2. 설계와 방법론 / 20점

상태: 강함

문서 반영 위치:

- 작품설명서: `3. 설계와 방법론`, `4. AI와 데이터 활용`

강점:

- `오답 기록 -> AI 진단 -> 같은 개념 재도전 -> 개선 확인` 흐름이 명확하다.
- `attempt_log`, `concept_summary`, `demo_seed`, `judge_demo`, `human_trial` 데이터 구분이 들어갔다.
- 개인 참가자 기준 개발 일정이 정리되어 있다.

남은 리스크:

- 발표에서 데이터 필드 이름만 말하면 어려워 보일 수 있다. 건호는 "틀린 기록표"와 "개념별 약점표"처럼 쉽게 풀어 말해야 한다.

## 3. 구현 및 유용성 / 20점

상태: 강함

문서 반영 위치:

- 작품설명서: `6. 구현 및 유용성`, `7. 작품 테스트`

강점:

- 초6 수학 대표문항 6개, 30초 데모, 대시보드, 데이터 화면이 구현되어 있다.
- `npm test` 기준 44 PASS / 0 FAIL이다.
- Manus API가 실패해도 fallback으로 흐름이 유지된다.

남은 리스크:

- 실제 HWP 제출 전에는 화면 시연을 한 번 더 사람이 직접 확인해야 한다. 현재는 Auto-Verified이고 Human-Verified는 아직 아니다.

## 4. 창의성 / 20점

상태: 강함

문서 반영 위치:

- 작품요약서: 차별점 문단
- 작품설명서: `5. 유사 제품 및 차별점`, `8. 창의성`

강점:

- 일반 문제풀이 앱과 달리 오답 원인, 오개념 데이터, 재도전 성공 여부를 연결했다.
- "오답 DNA"라는 이름이 기능과 잘 맞는다.
- AI를 단순 문제 생성기가 아니라 데이터와 연결된 오답 코치로 썼다.

남은 리스크:

- 친구 기능, XP, 꾸미기 요소가 심사 핵심처럼 보이면 점수가 흐려진다. 발표에서는 오답 데이터와 AI 진단을 먼저 보여준다.

## 5. 제출 자료 / 20점

상태: 보통에서 강함

문서 반영 위치:

- 작품설명서: `9. 제출 자료와 출처`, `10. 위험성 검토`
- 붙여넣기 가이드: 심사기준 체크

강점:

- 작품요약서와 작품설명서가 같은 메시지로 정리되었다.
- 출처와 참고 기준, 위험성 검토가 들어갔다.
- 화면에서 데이터 QA와 그래프/분포가 보이도록 설명했다.

남은 리스크:

- HWP에 붙여 넣은 뒤 표, 줄바꿈, 그래픽 캡처가 빠지면 제출 자료 점수가 깎일 수 있다.

## 최종 제출 전 3개 체크

1. 건호가 30초 데모를 직접 실행하고 "오답 기록 -> AI 진단 -> 재도전 -> 개선률"을 말할 수 있어야 한다.
2. 작품설명서에 화면 캡처를 넣는다면 `오답 DNA 지도`, `오답 원인 분포`, `데이터 QA` 화면을 우선 넣는다.
3. Human-Verified라고 쓰려면 `GEONHO_HUMAN_TRIAL_CHECKLIST.md`를 실제로 채워야 한다.
